Makefile: documentação executável

Escrevendo receitas para projetos uniformes, didáticos e evolutivos

Fernando Machado
3 min read
·
Jul 30, 2020

Sempre que começo a trabalhar em uma codebase nova, uma das primeiras perguntas que me vem a mente é: "como é que eu quebro testes aqui?".

Dependendo da linguagem ou do framework utilizados, comandos radicalmente diferentes devem ser utilizados para rodar testes. Além disso, muitas vezes, outros detalhes como quais testes devem ser rodados ou onde iremos armazenar os resultados da cobertura de testes também devem ser informados no comando.

Para facilitar a vida de quem chega (e também de nós mesmos no futuro), um time pode documentar esses comandos no README.md do projeto. Entretanto, como esse arquivo nunca é executado, essa prática acaba sendo tão efetiva quanto escrever o que um algoritmo deveria fazer nos comentários do código: cedo ou tarde, o algoritmo vai mudar e o comentário tem grandes chances de se tornar obsoleto.

Escrevendo receitas executáveis

A ferramenta Make, nativa em sistemas operacionais como Linux e MacOS, foi criada em 1976 com o propósito de automatizar a compilação de código-fonte em arquivos executáveis. As instruções para realizar essa tarefa são organizadas em um arquivo de texto chamado Makefile.

Ao reunir os comandos utilizados para compilar, testar e rodar a aplicação em um arquivo Makefile na raiz do projeto, o time começa a manter uma documentação executável que, por ser usada frequentemente, corre menos riscos de se tornar obsoleta sem que alguém perceba.

Quando substituímos o comando usado para rodar testes em Go:

go test ./... -v -covermode=count

por

make test

estamos expondo uma interface mais enxuta, tornando explícito apenas os comandos úteis para quem vai desenvolver naquela base de código. Além disso, ao encapsular a complexidade desnecessária, tornamos nosso projeto mais didático e facilitamos o onboarding de quem estiver chegando no time.

Nós podemos escrever instruções no Makefile para:

  • help: listar comandos disponíveis
  • setup: baixar e instalar módulos e dependências
  • format: aplicar regras de formatação (com linters)
  • build: compilar binário executável (ou transpilar Javascript)
  • test: executar testes disponíveis
  • run: iniciar a aplicação

…além de outras necessidades que podem surgir de acordo com o domínio ou a linguagem, como comandos para definir as variáveis de ambiente e subir um banco de dados ou um servidor de mensageria localmente.

Já em projetos fullstack, onde precisamos interagir com bases de código de frontend e backend potencialmente escritas em linguagens diferentes, nossa memória vai agradecer por poder executar sempre um mesmo comando agnóstico ao invés de ter que se lembrar quais são os comandos e argumentos necessários para subir as diferentes aplicações.

Essa uniformidade na forma de interagir com o código também nos ajuda a garantir que, tanto as pessoas que trabalham nele quanto as ferramentas usadas para integração e distribuição contínua falem a mesma língua. Conforme nossa aplicação evolui, os envolvidos no projeto tem acesso imediato aos novos comandos. Assim, caso algum problema aconteça em alguma etapa do desenvolvimento, fica mais fácil tentar reproduzir o erro.

Confira um exemplo de como um Makefile pode ser organizado:

.SILENT: indica qual comando deve ser executado quando nenhum argumento é informado. Nesse caso, o Make irá exibir as comandos disponíveis em help.

.PHONY: marcamos os comandos que criamos como .PHONY para deixar explícito que, ao contrário do que sugerem as origens da ferramenta, ele não será necessariamente utilizado para construir um arquivo (trívia: “phony” é uma expressão em inglês usada para definir “algo que não é o que parece ser”)

Essa é apenas uma sugestão pra servir de inspiração para novas receitas. Arquivos diferentes deverão ser criados para resolver problemas diferentes. Se você usa Makefile em seu projeto, escreva nos comentários quais comandos estão sendo úteis dentro do seu time. Se não, que tal jogar esse artigo no chat do projeto pra ver o que sai? :)
